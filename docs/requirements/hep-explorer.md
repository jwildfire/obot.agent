# hep-explorer requirements matrix

> Source-backed requirement matrix for the safety.viz **hep-explorer** (eDISH) module, harvested from the original [SafetyGraphics/hep-explorer](https://github.com/SafetyGraphics/hep-explorer) renderer (v1.4.2) cross-referenced with the safety.viz port, plus the baseline-referenced **composite plot** (`HEP-COMP-*`, [safety.viz#67](https://github.com/jwildfire/safety.viz/issues/67)) and the abnormal-baseline **migration (Sankey) view** together with the shared hepatic domain layer it extracts (`HEP-CORE-*`, `HEP-MIG-*`, `HEP-XTAB-*`, `HEP-STEP-*`, `HEP-ARM-*`, `HEP-ACC-*`, [obot.roadmap#43](https://github.com/jwildfire/obot.roadmap/issues/43)). Treat this as a reviewed starting matrix: the shipped rows are AI-reviewed and awaiting human sign-off; the deferred rows are filed as v1.3.0 follow-up sub-issues; the `planned` rows are authored ahead of implementation and carry no evidence yet. Requirement IDs use the module's condensed `HEP-*` scheme, matching the IDs cited in the source JSDoc, the tests, and the port's [`docs/hep-explorer-coverage.md`](https://github.com/jwildfire/safety.viz/blob/main/docs/hep-explorer-coverage.md). IDs are the **base** form (`HEP-QUAD-001`, not `-001A`/`-001B`) so each row's ID resolves the matching `requirementIds` tag in the shipped `docs/evidence/hep-explorer/evidence.json`, which the evidence pages look up by exact ID; where a source requirement was split during review, the atomic parts are merged into the single base-ID row. The `Evidence Type` column routes rows: `unit` → Vitest (`tests/unit/hep-explorer/`), `browser` → Playwright (`tests/e2e/hep-explorer.spec.js`), `deferred` → filed as a v1.3.0 sub-issue (not yet implemented). Where a source row is covered by both suites, the browser evidence is listed. For a row whose Status is `planned`, the `Evidence Type` names the suite that *will* evidence it and the `Test/Evidence Link` points at the implementation issue until `docs/hep-explorer-coverage.md` lists it.

## Composite plot (HEP-COMP)

The baseline-referenced **composite plot** (safety.viz [#67](https://github.com/jwildfire/safety.viz/issues/67), PR [safety.viz#69](https://github.com/jwildfire/safety.viz/pull/69)) is a faithful port of the FDA Composite-eDISH-Plot reference (`Composite_eDISH_Model.R`, public domain / MIT, DOI [10.5281/zenodo.10892050](https://doi.org/10.5281/zenodo.10892050)) implementing Tesfaldet et al., "Composite Plot for Visualizing Aminotransferase and Bilirubin Changes in Clinical Trials of Subjects with Abnormal Baseline Values," Drug Safety 2024;47:699–710. ## Abnormal-baseline migration view (HEP-CORE, HEP-MIG, HEP-XTAB, HEP-STEP, HEP-ARM, HEP-ACC)

The **migration view** (`view: 'migration'`, label "Migration (Sankey)") is the module's third view and implements Figure 3 of Amirzadegan et al., Drug Safety 2025;48:443–453 (DOI 10.1007/s40264-024-01511-8, PMID 39932652, PMC11982145): a bidirectional Sankey of each participant's shift from their pretreatment eDISH quadrant to their maximum on-treatment quadrant, mirrored about one shared baseline column so the placebo arm reads left and the active arm reads right, with the paper's per-arm cross tables beside it. It sits between the scatter and composite views and hands a selected flow to the composite plot, which is the paper's own two-step review made executable: the Sankey raises the question at population level, the ×Baseline composite answers it for the named participants. The `HEP-CORE-*` rows cover the shared hepatic domain layer (`src/hep-core/`) extracted from `src/hep-explorer.js` and `src/hep-explorer/composite.js` ahead of that work — one cohort, one baseline definition and one on-treatment definition for Figures 3, 4 and 5 — so the sibling `hep-waterfall` module inherits them rather than re-deriving them.

Three notes on how these rows are written. **(1)** They are authored *before* implementation ([obot.roadmap#43](https://github.com/jwildfire/obot.roadmap/issues/43)) so the evidence pages render reviewed requirement text from the very first run instead of degrading to IDs-only; every row therefore carries Status `planned` and links its implementation issue until `docs/hep-explorer-coverage.md` lists it. **(2)** The accessibility rows are `HEP-ACC-*`, **not** `HEP-A11Y-*`: the requirement-ID grammar `^[A-Z]{2,4}-[A-Z]+-\d+[A-D]?$` (`scripts/requirements-lib.mjs`, mirrored unanchored in `scripts/evidence-lib.mjs`) forbids digits in the area segment, so an `A11Y` ID would parse as no requirement at all — silently absent from the extract and unresolvable on the evidence page. **(3)** `HEP-DATA-011` and `HEP-DATA-012` extend the module's existing `HEP-DATA-*` series but are listed here, with the workstream that adds them, rather than in the main table above.
