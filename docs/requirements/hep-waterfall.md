# hep-waterfall requirements matrix

> Source-backed requirement matrix for the safety.viz **hep-waterfall** (modified ALT waterfall) module. Unlike every other matrix in this directory, hep-waterfall is **not a port of a legacy RhoInc/SafetyGraphics renderer** — it is a new renderer derived from a published FDA paper: Amirzadegan et al., *Drug Safety* 2025;48:443–453 ([DOI 10.1007/s40264-024-01511-8](https://doi.org/10.1007/s40264-024-01511-8), [PMID 39932652](https://pubmed.ncbi.nlm.nih.gov/39932652/), [PMC11982145](https://pmc.ncbi.nlm.nih.gov/articles/PMC11982145/)), **Figure 5** — the modified waterfall for trials enrolling participants with abnormal baseline liver tests. The `Source` column therefore cites the paper's figure caption, its Table 1 applicability rule, and its body text — or the design that resolves them ([obot.roadmap#43 design](https://jwildfire.github.io/obot.roadmap/requirements/design/43_design.html)) — rather than a source repository. Requirement IDs use the module's condensed `HWF-*` scheme and are the **base** form, so each row resolves the matching `requirementIds` tag in the shipped `docs/evidence/hep-waterfall/evidence.json`, which the evidence pages look up by exact ID. The `Evidence Type` column states the suite each row is routed to: `unit` → Vitest (`tests/unit/hep-waterfall/`), `browser` → Playwright (`tests/e2e/hep-waterfall.spec.js`); where a row will be covered by both suites the browser evidence is listed. This matrix is authored **before** implementation ([safety.viz#93](https://github.com/jwildfire/safety.viz/issues/93)) so that `npm run requirements` publishes real requirement text rather than degrading the evidence page to IDs-only; until that PR merges the `Test/Evidence Link` column points at the implementing issue and is re-pointed at `docs/hep-waterfall-coverage.md` when the coverage doc ships. Treat this as a drafted matrix: the rows are AI-drafted and awaiting human sign-off, and **three rows are marked `needs-jeremy-review`** (`HWF-ORDER-002`, `HWF-ORDER-003`, `HWF-BOX-003`) because the paper's own caption is either self-contradictory (participant ordering) or silent (the flanking box panels) — see Scope below.

## Requirement context

- Requirement: [obot.roadmap#43](https://github.com/jwildfire/obot.roadmap/issues/43) — abnormal-baseline DILI tools (Figures 3 and 5 of Amirzadegan et al. 2025)
- Design: [`requirements/design/43_design.html`](https://jwildfire.github.io/obot.roadmap/requirements/design/43_design.html) — section references in the `Source` column (§E waterfall design, §F data contract, §G demo data, §K open calls) are to this document
- Implementation, a stack of three PRs: [safety.viz#91](https://github.com/jwildfire/safety.viz/issues/91) (PR1 — `src/hep-core/` extraction and the `hep-explorer` view split this module's per-participant reduction comes from), [safety.viz#92](https://github.com/jwildfire/safety.viz/issues/92) (PR2 — Figure 3, the migration Sankey), [safety.viz#93](https://github.com/jwildfire/safety.viz/issues/93) (PR3 — **this module**, end to end)
- The shared hepatic domain layer (`HEP-CORE-*`) and Figure 3 (`HEP-MIG-*`, `HEP-XTAB-*`, `HEP-STEP-*`, `HEP-ARM-*`, `HEP-A11Y-*`) are **not** in this matrix — they live in [`hep-explorer.md`](./hep-explorer.md), because Figure 3 is a third view of `hep-explorer` rather than a module. This matrix covers only what is specific to the waterfall.

## Scope

**What the module is.** One floating bar per participant, spanning from that participant's own baseline ALT to their maximum on-treatment ALT, in **absolute reporting units (U/L)** rather than multiples of the upper limit of normal or of baseline; participants ranked by baseline value so a continuous black baseline trace rises across the plot; the placebo arm filling the left half and the active arm the right half, mirrored about a centre divider with a shared vertical scale drawn on both flanks; bars coloured blue (placebo) and bronze (active) with green overriding both for participants who develop new-onset jaundice; and a summary box-and-whisker panel flanking each side. Total bilirubin is used **only** as a ×ULN quantity — for the cohort rule and the jaundice flag — and is never plotted on the absolute axis.

**Why it is a separate renderer and not another `hep-explorer` view — the paper's Table 1 applicability rule.** Table 1 assigns this tool to a *different population* from the eDISH scatter, the migration Sankey and the composite plot: trials enrolling participants with **elevated baseline ALT but normal baseline bilirubin**, and it **excludes baseline-jaundiced participants** — precisely the participants the migration and composite views require. Different population, different units (absolute U/L rather than ×ULN), different question. That exclusion is not a footnote but a plotted behaviour: `HWF-DATA-003` requires the excluded count to be reported on the page, and the demo cohort deliberately contains participants that fail the rule so the exclusion is demonstrable evidence rather than a claim.

**Two rows rest on a judgement call a human reviewer must check against the published figure.** Both are recorded as open calls in the design (§K) and are marked `needs-jeremy-review` in the table:

1. **Participant ordering (`HWF-ORDER-002`, `HWF-ORDER-003`; design §E.3, open call O3).** The Figure 5 caption contradicts itself: read literally, "active drug subjects run right-to-center highest-to-lowest" places the *lowest* active baselines at the centre, which contradicts its own controlling clause that "the highest baselines meet in the center". The design takes the latter as intent and pins it as a testable invariant — a unimodal baseline trace peaking at the arm boundary. If the published figure shows the literal reading instead, `HWF-ORDER-002` and `HWF-ORDER-003` both invert. `HWF-ORDER-001` (ascending placebo, left to right) is uncontested and holds either way.
2. **Two boxes per flanking panel (`HWF-BOX-003`; design §E.6, open call O2).** The caption says only "summary box-and-whisker plots, one per arm" and does not state how many boxes each panel carries. The design reads the panel's job as summarising the *shift* — the arm-level answer to the question the bars answer per participant — and therefore draws a **baseline** box and a **maximum on-treatment** box in each panel, with a `summary: 'peak'` setting exposing the single-box alternative. Confirm against the published figure before the guide is written.

**Out of scope / recorded costs.** Selection does not carry between `hep-waterfall` and `hep-explorer` — the two modules serve different trials, and a cross-link is all the relationship they get (design §K R6). The demo runs on a purpose-built synthetic cohort, `site/data/adbds-abnbl.csv` (`ABL-` prefix), because the shipped demo file cannot express the figure: verified, after the mandated baseline-bilirubin exclusion **zero** participants in `site/data/adbds.csv` have baseline ALT ≥ 3×ULN, and the whole file contains **two** new-onset-jaundice participants, one per arm — which contradicts the paper's "several developed jaundice" on active drug. Demoing a chart on data authored for it is close to circular and is recorded as open call O4, not pretended away. The module ships `experimental: true` (open call O7) so its settings surface is not frozen while O2, O3 and O4 are open.
